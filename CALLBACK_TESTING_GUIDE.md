## Quick Start

The easiest way to test is using the provided PHP script:

```bash
php test_callback.php [scenario]
```

Available scenarios:
- `valid` - Valid callback with existing transaction (default)
- `approved` - Test approved status
- `declined` - Test declined status with error
- `processing` - Test processing status
- `invalid-signature` - Test with invalid signature
- `missing-fields` - Test with missing required fields
- `no-transaction` - Test with non-existent transaction
- `duplicate` - Test duplicate callback (idempotency)

## Using the Test Script

### Prerequisites
1. Ensure you have at least one user in the database
2. Ensure you have at least one transaction (or the script will create one)
3. Set your `.env` file with correct configuration

### Basic Usage

```bash
# Test with valid callback
php test_callback.php valid

# Test approved status
php test_callback.php approved

# Test declined status
php test_callback.php declined

# Test invalid signature
php test_callback.php invalid-signature

# Test missing fields
php test_callback.php missing-fields

# Test non-existent transaction
php test_callback.php no-transaction
```

### What the Script Does

1. **Finds or Creates Test Transaction**: Looks for an existing transaction or creates one
2. **Generates Callback Payload**: Creates a properly formatted callback payload
3. **Signs the Payload**: Uses the same signing algorithm as the payment gateway
4. **Sends the Callback**: POSTs to your callback endpoint
5. **Displays Results**: Shows HTTP status, response, and transaction updates

## Manual Testing with cURL

### Step 1: Create a Test Transaction

First, create a transaction through your application or use the database:

```sql
INSERT INTO transactions (user_id, merchant_order_id, order_id, operation_id, operation_type, amount, currency, status, pay_method, order_desc, merchant_id, created_at, updated_at)
VALUES (1, 'test-order-123', 'c1234567-1234-1234-1234-123456789abc', 'f1234567-1234-1234-1234-123456789abc', 'payout', 100.00, 'USD', 'processing', 'card', 'Test transaction', 85, NOW(), NOW());
```

### Step 2: Generate Signed Callback

Use the test script to generate a signed callback, or manually create one:

```php
<?php
require 'vendor/autoload.php';
$dotenv = Dotenv\Dotenv::createImmutable(__DIR__);
$dotenv->load();

use App\Services\PaymentServiceProvider;

$provider = new PaymentServiceProvider(env('PAYMENT_MERCHANT_CONTROL_VALUE'));

$callbackData = [
    'merchant_order_id' => 'test-order-123',
    'order_desc' => 'Test callback',
    'amount' => '100.00',
    'currency' => 'USD',
    'merchant' => [
        'id' => 85,
        'custom_data' => new stdClass(),
    ],
    'order_id' => 'c1234567-1234-1234-1234-123456789abc',
    'operation_id' => 'f1234567-1234-1234-1234-123456789abc',
    'operation_type' => 'payout',
    'status' => 'approved',
    'pay_method' => 'card',
    'created_at' => '2024-01-01T12:00:00+0000',
    'finished_at' => '2024-01-01T12:00:00+0000',
];

$json = json_encode($callbackData, JSON_UNESCAPED_SLASHES);
$signed = $provider->sign($json);
echo $signed;
```

### Step 3: Send cURL Request

```bash
curl -X POST http://localhost:8000/api/payment/callback \
  -H "Content-Type: application/json" \
  -H "Accept: application/json" \
  -d '{
    "signature": "740e15e5ba066c37f47e322627175d351b00eacf0908d2ca05d3837c38f7c21b",
    "merchant_order_id": "test-order-123",
    "order_desc": "Test callback",
    "amount": "100.00",
    "currency": "USD",
    "merchant": {
      "id": 85,
      "custom_data": {}
    },
    "order_id": "c1234567-1234-1234-1234-123456789abc",
    "operation_id": "f1234567-1234-1234-1234-123456789abc",
    "operation_type": "payout",
    "status": "approved",
    "pay_method": "card",
    "created_at": "2024-01-01T12:00:00+0000",
    "finished_at": "2024-01-01T12:00:00+0000"
  }'
```

**Note**: Replace the signature with one generated by your `PaymentServiceProvider`.

## Testing with Postman

### Setup

1. **Create a new POST request**
   - URL: `http://localhost:8000/api/payment/callback`
   - Method: `POST`

2. **Set Headers**
   ```
   Content-Type: application/json
   Accept: application/json
   ```

3. **Create Request Body** (raw JSON)

Use the test script to generate a valid signed payload, then copy it to Postman.

## Testing Scenarios

### 1. Valid Callback

**Expected Result**: 
- HTTP 200
- Transaction status updated
- Success message in response

```bash
php test_callback.php valid
```

### 2. Approved Status

**Expected Result**:
- HTTP 200
- Transaction status = "approved"
- finished_at timestamp set

```bash
php test_callback.php approved
```

### 3. Declined Status

**Expected Result**:
- HTTP 200
- Transaction status = "declined"
- error_message and error_code set
- finished_at timestamp set

```bash
php test_callback.php declined
```

### 4. Invalid Signature

**Expected Result**:
- HTTP 200 (to prevent retries)
- Error logged
- Transaction NOT updated
- Response indicates invalid signature

```bash
php test_callback.php invalid-signature
```

### 5. Missing Required Fields

**Expected Result**:
- HTTP 200 (to prevent retries)
- Error logged
- Response indicates missing fields

```bash
php test_callback.php missing-fields
```

### 6. Non-Existent Transaction

**Expected Result**:
- HTTP 200 (to prevent retries)
- Warning logged
- Response indicates transaction not found

```bash
php test_callback.php no-transaction
```

### 7. Duplicate Callback (Idempotency)

**Expected Result**:
- HTTP 200
- Duplicate detected
- Transaction NOT updated (idempotent)
- Response indicates duplicate

**Steps**:
1. Run a valid callback first
2. Run the same callback again immediately

```bash
php test_callback.php valid
php test_callback.php duplicate
```

## Verifying Results

### 1. Check HTTP Response

All callbacks should return HTTP 200 with a JSON response:

```json
{
  "success": true,
  "message": "Callback processed"
}
```

### 2. Check Logs

**Callback Logs**:
```bash
tail -f storage/logs/callbacks/callbacks-$(date +%Y-%m-%d).log
```

**Transaction Logs**:
```bash
tail -f storage/logs/transactions/transactions-$(date +%Y-%m-%d).log
```

**API Logs**:
```bash
tail -f storage/logs/api/api-$(date +%Y-%m-%d).log
```

### 3. Check Database

Verify transaction was updated:

```sql
SELECT id, merchant_order_id, status, error_message, error_code, finished_at, updated_at
FROM transactions
WHERE merchant_order_id = 'test-order-123'
ORDER BY updated_at DESC;
```

### 4. Check Application Logs

```bash
tail -f storage/logs/laravel.log
```

## Testing on Production/Staging

### Using ngrok (for local development)

1. **Install ngrok**: https://ngrok.com/

2. **Start your Laravel server**:
   ```bash
   php artisan serve
   ```

3. **Start ngrok**:
   ```bash
   ngrok http 8000
   ```

4. **Update callback URL** in payment gateway dashboard:
   ```
   https://your-ngrok-url.ngrok.io/api/payment/callback
   ```

5. **Test with real payment gateway**:
   - Create a test transaction
   - Wait for callback
   - Check logs and database

### Using Production URL

1. **Set callback URL** in payment gateway:
   ```
   https://yourdomain.com/api/payment/callback
   ```

2. **Ensure SSL is configured** (required for production)

3. **Test with real transactions**:
   - Create test transactions
   - Monitor logs
   - Verify callbacks are received

## Troubleshooting

### Callback Not Received

1. **Check firewall/security settings**
2. **Verify callback URL is accessible**:
   ```bash
   curl -X POST https://yourdomain.com/api/payment/callback
   ```
3. **Check server logs** for incoming requests
4. **Verify CSRF is disabled** for callback route

### Invalid Signature Errors

1. **Verify `PAYMENT_MERCHANT_CONTROL_VALUE`** matches payment gateway
2. **Check signature calculation** using test script
3. **Review callback payload structure**

### Transaction Not Found

1. **Verify transaction exists** in database
2. **Check `merchant_order_id`** matches
3. **Verify `order_id`** if provided

### Duplicate Callbacks

This is normal behavior. The system handles duplicates idempotently:
- First callback: Updates transaction
- Duplicate callback: Acknowledged but not updated

## Best Practices

1. **Always test in staging first**
2. **Monitor logs** during testing
3. **Test all scenarios** (approved, declined, processing)
4. **Test idempotency** with duplicate callbacks
5. **Verify signature verification** works correctly
6. **Check error handling** for edge cases
7. **Monitor production callbacks** regularly
